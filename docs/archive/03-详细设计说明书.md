## 3. 各模块详细设计

### 3.1 用户认证模块

#### 3.1.1 类设计

| 类名 | 职责 |
|------|------|
| AuthController | 提供登录、注册、登出、刷新Token接口 |
| AuthService | 处理认证业务逻辑 |
| JwtTokenProvider | JWT Token的生成与解析 |
| JwtAuthenticationFilter | 拦截请求并校验Token |
| SecurityConfig | Spring Security配置类 |
| CustomUserDetailsService | 加载用户详情及权限 |

#### 3.1.2 时序图（登录流程）

```
用户 -> 前端: 输入用户名密码
前端 -> AuthController: POST /api/auth/login
AuthController -> AuthService: authenticate(username, password)
AuthService -> CustomUserDetailsService: loadUserByUsername()
CustomUserDetailsService -> UserRepository: findByUsername()
UserRepository --> CustomUserDetailsService: UserEntity
CustomUserDetailsService --> AuthService: UserDetails
AuthService -> PasswordEncoder: matches(raw, encoded)
AuthService -> JwtTokenProvider: generateToken(userDetails)
JwtTokenProvider --> AuthService: AccessToken + RefreshToken
AuthService --> AuthController: TokenDTO
AuthController --> 前端: 200 + TokenDTO
前端 -> 前端: 存储Token，跳转首页
```

## 4. 关键算法设计

### 4.1 JWT 认证流程

```java
// 1. 登录验证成功后生成 Token
String accessToken = Jwts.builder()
    .subject(user.getUsername())
    .claim("roles", roles)
    .claim("userId", user.getId())
    .issuedAt(new Date())
    .expiration(new Date(System.currentTimeMillis() + ACCESS_TOKEN_EXPIRE))
    .signWith(secretKey)
    .compact();

// 2. 请求拦截时解析 Token
Claims claims = Jwts.parser()
    .verifyWith(secretKey)
    .build()
    .parseSignedClaims(token)
    .getPayload();
```

### 4.2 逾期罚金计算算法

```
输入：returnDate, dueDate, dailyFineRate
输出：fineAmount

if returnDate <= dueDate:
    fineAmount = 0
else:
    overdueDays = daysBetween(dueDate, returnDate)
    fineAmount = overdueDays * dailyFineRate
    // 设置罚金上限
    fineAmount = min(fineAmount, MAX_FINE_AMOUNT)
```

### 4.3 库存扣减并发控制

```java
@Modifying
@Query("UPDATE Book b SET b.availableStock = b.availableStock - 1 " +
       "WHERE b.id = :bookId AND b.availableStock > 0")
int decreaseStock(@Param("bookId") Long bookId);

// 使用数据库乐观锁/原子更新防止超卖/超借
if (affectedRows == 0) {
    throw new BusinessException("库存不足");
}
```

库存相关 Service 必须加事务边界。审批通过、取书、归还、取消预约等流程要同时写入业务表与 `book_stock_log`，任一写入失败必须整体回滚。

## 6. 异常处理机制

### 6.1 异常分类

| 异常类型 | 处理策略 |
|----------|----------|
| 业务异常(BusinessException) | 返回明确的错误码和提示信息 |
| 参数校验异常(ValidationException) | 返回字段级校验错误详情 |
| 认证异常(AuthenticationException) | 返回401，前端引导重新登录 |
| 权限异常(AccessDeniedException) | 返回403，前端提示无权限 |
| 系统异常(RuntimeException) | 返回500，记录错误日志 |

### 6.2 统一异常处理示例

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return Result.fail(400, msg);
    }
}
```

## 7. 开发完成状态与后续建议

当前项目核心功能已基本实现完成，后续迭代建议按以下优先级进行：

| 优先级 | 工作内容 | 验收点 |
|--------|----------|--------|
| P1 | 测试补全：Service 层单元测试、核心接口集成测试、并发借阅/库存测试 | 核心 Service 覆盖率 ≥ 70%，借阅并发场景通过 |
| P2 | 性能与安全：接口性能基线、SQL 注入/XSS/越权安全测试 | 200 并发平均响应时间 < 2s，错误率 < 1% |
| P3 | 小程序体验增强：ISBN 扫码查书、消息通知、UI 细节优化 | 扫码可识别图书，页面无明显体验问题 |
| P4 | 运维增强：监控告警、日志聚合、备份自动化 | Prometheus/Grafana 或同类方案可观测 |

历史开发顺序（供参考）：后端基础设施 → 数据库迁移与实体 → 管理端基础 → 图书分类与图书管理 → 小程序登录与首页/搜索/详情 → 借阅审批/归还/续借/逾期 → 预约/公告/统计/日志 → Docker 部署与测试。

## 8. Agent 实现细则

1. 后端每个模块至少包含 `entity`、`repository`、`service`、`controller`、`dto`，复杂返回对象放入 `vo`。
2. 请求 DTO 必须使用 Bean Validation 注解，Controller 参数必须触发校验。
3. 不在 Controller 中直接访问 Repository，不在前端组件中直接散写 `fetch` 或 `axios`。
4. 管理端页面应优先使用 Element Plus 表格、表单、弹窗和分页；小程序端页面应使用 Uni-App 兼容组件。
5. 所有状态枚举在后端使用枚举类表达，前端维护统一字典映射，避免页面散落魔法数字。
6. 每个功能完成后至少补充成功、失败、无权限三类测试或验证记录。
