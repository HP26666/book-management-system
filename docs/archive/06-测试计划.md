## 2. 测试范围

### 2.1 纳入测试范围

- 用户认证与权限管理模块
- 图书信息管理模块
- 图书分类管理模块
- 借阅管理模块
- 预约管理模块

## 3. 测试策略

### 3.1 单元测试

| 项目 | 说明 |
|------|------|
| 测试对象 | 后端Service层、工具类、Mapper转换器 |
| 测试框架 | JUnit 5 + Mockito |
| 覆盖率目标 | 核心业务逻辑覆盖率 ≥ 70% |
| 执行时机 | 每次代码提交前本地执行，CI流水线自动执行 |

#### 重点测试类

- `AuthService`：登录认证、Token生成
- `BookService`：图书CRUD、库存校验
- `BorrowService`：借阅申请、审批、归还、逾期计算
- `ReaderService`：读者信息维护、借阅额度校验

### 3.2 接口测试

| 项目 | 说明 |
|------|------|
| 测试对象 | 所有RESTful API接口 |
| 测试工具 | Postman / Apifox |
| 测试内容 | 正常请求、参数校验、权限控制、异常响应 |
| 执行时机 | 后端开发完成后进行，回归测试时复用 |

#### 接口测试重点

- 登录接口：正确/错误密码、Token有效性
- 图书接口：CRUD操作、分页查询、权限隔离
- 借阅接口：并发借阅下的库存扣减正确性
- 文件上传接口：文件类型限制、大小限制

### 3.3 集成测试

| 项目 | 说明 |
|------|------|
| 测试对象 | 前后端联调、数据库事务一致性、Redis缓存一致性 |
| 测试环境 | 独立的测试服务器 + 测试数据库 |
| 测试内容 | 完整的业务流程端到端验证 |

#### 集成测试场景

1. **完整借阅流程**：用户登录 → 搜索图书 → 提交借阅 → 管理员审批 → 用户查看借阅状态 → 管理员登记归还
2. **预约释放流程**：图书无库存 → 用户预约 → 图书归还 → 系统通知预约用户 → 用户借阅/预约过期释放
3. **权限隔离验证**：普通用户尝试访问管理员接口，系统返回403

### 3.4 系统测试

| 项目 | 说明 |
|------|------|
| 测试对象 | 完整的系统功能与用户体验 |
| 测试环境 | 与生产环境一致的UAT环境 |
| 测试内容 | 功能完整性、数据准确性、界面友好性 |

### 3.5 性能测试

| 项目 | 说明 |
|------|------|
| 测试工具 | JMeter / k6 |
| 测试场景 | 图书搜索接口并发查询、借阅申请并发提交 |
| 目标指标 | 200并发下，平均响应时间 < 2s；错误率 < 1% |

### 3.8 部署验收测试

| 测试项 | 验证方式 |
|--------|----------|
| Compose 构建 | 执行 `docker compose up -d --build`，所有容器状态为 healthy 或 running |
| 管理端访问 | 访问 `http://localhost`，页面正常加载且静态资源无 404 |
| 后端健康 | 访问 `http://localhost:8080/actuator/health` 返回 UP |
| API 代理 | 访问 `http://localhost/actuator/health` 能通过 Nginx 代理后端 |
| 环境变量 | 修改 `.env` 端口后重新启动，端口映射符合预期 |

### 3.6 兼容性测试

| 测试端 | 测试内容 |
|--------|----------|
| PC Web管理端 | Chrome 120+、Edge 120+、Firefox 120+ |
| 微信小程序端 | 微信最新版、前两个主版本 |
| 移动端适配 | iPhone SE / iPhone 15 / 主流Android机型 |

### 3.7 安全测试

| 测试项 | 测试内容 |
|--------|----------|
| 认证安全 | Token过期验证、暴力登录限制 |
| 权限安全 | 水平越权（访问他人数据）、垂直越权（普通用户访问管理员接口） |
| 输入安全 | SQL注入、XSS攻击、特殊字符过滤 |
| 接口安全 | 敏感接口未授权访问、文件上传路径遍历 |

## 6. 测试用例设计方法

- **等价类划分法**：用于输入参数校验场景（如用户名长度、手机号格式）
- **边界值分析法**：用于分页查询、库存数量、借阅天数等边界场景
- **场景法**：用于借阅、预约、归还等完整业务流程
- **错误推测法**：用于权限越权、并发冲突、Token失效等异常场景

## 7. 风险与应对措施

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 开发延期导致测试时间压缩 | 高 | 提前介入接口评审，接口测试与开发并行 |
| 测试数据不足 | 中 | 准备脚本批量生成模拟数据 |
| 第三方ISBN接口不稳定 | 低 | 测试环境中使用Mock数据 |
| 并发性能不达标 | 中 | 预留1周性能优化时间 |

---

## 8. 测试通过标准

1. **功能标准**：所有高优先级用例100%通过，中优先级用例通过率 ≥ 95%
2. **缺陷标准**：无严重（Critical）缺陷，一般（Major）缺陷 ≤ 5个且均有规避方案
3. **性能标准**：核心接口在目标并发下平均响应时间达标，错误率 < 1%
4. **安全标准**：无高危安全漏洞，中危漏洞已修复或有防护措施
5. **文档标准**：测试报告完整，缺陷记录清晰可追溯

---

## 9. Agent 测试补充要求

1. 后续每完成一个需求编号，必须补齐对应的单元测试、接口测试或手工验收记录。
2. 借阅、归还、预约、库存调整必须包含并发和事务回滚测试。
3. 权限测试必须覆盖未登录、普通读者、图书管理员、系统管理员四类身份。
4. 管理端和小程序端至少各保留一组核心流程端到端测试或可复现手工脚本。
5. Docker 验收失败时，不允许标记功能开发完成。
6. 自动化测试数据必须可重复初始化，不能依赖开发者本机已有数据。
